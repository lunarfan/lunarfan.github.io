# LunarFan 開發前規格文件（Phase 1）

## 1. 文件目的
本文件定義 LunarFan 第一階段（Phase 1）的開發範圍、技術選型與驗收標準。  
第一階段聚焦：**只做首頁（Landing Page）**。

## 2. 後端技術評估：NestJS vs Go

### 2.1 NestJS 優缺點
**優點**
- 開發速度快：TypeScript 生態完整，與 Next.js 同語言，開發與溝通成本低。
- 架構清晰：內建模組化、DI、Guard、Interceptor，適合快速建立可維護後端。
- 生態成熟：Auth、Validation、ORM、Swagger、Queue 等常見需求有完整套件。
- 團隊上手快：對熟悉 Node.js/TypeScript 的團隊非常友善。

**缺點**
- 效能與資源使用通常不如 Go（高併發下差異更明顯）。
- 運行時依賴 Node.js，對極致低延遲服務較不占優。

### 2.2 Go 優缺點
**優點**
- 高效能、低記憶體使用，適合高併發與高吞吐情境。
- 單一靜態編譯執行檔，部署簡潔、穩定性高。
- 原生併發模型（goroutine）適合 I/O 密集型服務。

**缺點**
- 初期開發速度通常慢於 NestJS（特別是後台管理、驗證、授權、文件整合）。
- 與前端不同語言棧，型別與 DTO 共享成本較高。
- 若團隊以 TS 為主，學習與協作成本相對增加。

### 2.3 結論（本專案）
- 以你目前目標（快速推進 MVP）來看，**Phase 1 選擇 NestJS**。
- 未來若成長到明顯高併發瓶頸，可再將特定服務拆分為 Go 微服務。

## 3. 技術架構
- 前端：Next.js + TypeScript
- 後端：NestJS + TypeScript
- DB：PostgreSQL
- 快取/Session/佇列：Redis（先完成基礎連線，進階用途可在後續階段啟用）
- 檔案儲存：S3 相容（開發 MinIO；正式環境可切 AWS S3）
- 金流：**綠界（ECPay）為優先選項**（Phase 1 不實作）
- 服務管理：Docker Compose

註記：以上為整體產品目標架構；**Phase 1 僅落地前端首頁**。

## 4. 第一階段開發範圍（In Scope）

### 4.1 基礎框架
- 建立前端專案（Next.js + TypeScript）
- 建立 Docker 化前端開發與建置流程（前端容器版本）
- 建立基本環境變數管理（`.env.example`）
- 建立 GitHub Pages 靜態輸出版本（短期展示用途）

### 4.2 首頁功能
- Header：
  - 左上顯示 `logo.png`（不額外顯示 `LunarFan` 字樣）
  - 右上未登入顯示「註冊 / 登入」
  - 右上登入後顯示 Avatar（可先用假資料狀態切換）
- 主內容：
  - 滾動吸附（scroll snap）兩段介紹區塊
  - 以大字標題與重點文案為核心（第一階段可先不放主內容圖片）
  - 第二段底部放置可點擊 CTA 文字
  - 效果參考 Webflow scroll-snap 範例風格
- Footer：
  - 顯示聯絡信箱 `contact.us@lunar.fan`

### 4.3 首頁文案定稿（Phase 1）
- 第一屏（置中大字）：
  - 主標：`LunarFan`（可搭配 Logo 一起呈現）
  - 副標：`遇見你最想追隨的創作者。`
- 第二屏（主題：為什麼選擇 LunarFan?）：
  - 區塊標題：`為什麼選擇 LunarFan?`
  - 右側插圖：`home_page_illustration.png`
  - 重點 1（對創作者）：`用最低的門檻建立後援會，操作起來就像一般社群媒體一樣順暢。`
  - 重點 2（對資深粉絲）：`可以建立私人後援會，與志同道合的夥伴盡情討論分享。`
  - 重點 3（對一般粉絲）：`即時掌握創作者的第一手消息，還有不定時的直播與專屬聊天室。`
  - 底部 CTA（可點擊）：`現在就加入 Go!`

### 4.4 視覺規範（Phase 1）
- 全站主底色：`#0A233E`
- Header logo：使用 `logo.png`（僅 logo 圖，不加品牌文字）
- 主要文字色：`#FFFFFF`
- 次要文字色：`rgba(255,255,255,0.75)`
- 第一屏主標以大字置中呈現，需優先確保首屏辨識度與對比
- 第二屏條列文案建議控制單行長度，維持桌機與手機可讀性

### 4.5 主內容圖片形象建議（Phase 1）
- 第一階段先以文字資訊與版面層次為主，不強制放置主內容圖片。
- 如後續補圖，需避免干擾第一屏大字與第二屏條列重點的可讀性。
- 補圖策略建議於 Phase 2 再定義（含風格、授權來源、壓縮與載入策略）。

## 6. 部署與發佈（Phase 1）
- 交付版本 A（短期）：GitHub Pages 靜態版
  - 用途：短期展示與暫時對外連結
  - 發佈方式：Next.js 靜態匯出後發佈到 GitHub Pages
  - 維護策略：僅短暫使用，後續停止更新
- 交付版本 B（正式）：Docker 容器版
  - 用途：後續正式環境與持續更新主版本
  - 發佈方式：以 Docker 容器部署（平台可為 VM / 雲端主機 / K8s）
  - 維護策略：GitHub Pages 下線後，由此版本完全接手

註記：GitHub Pages 不支援以 Docker 容器形式直接上線，因此兩版需分開產出。

### 6.1 版本切換策略
- 階段 1：先上 GitHub Pages 靜態版（快速展示）
- 階段 2：Docker 容器版上線並完成驗證
- 階段 3：凍結 GitHub Pages 內容，不再更新；對外流量切換至 Docker 版本
